Form Validator

The form validator is a powerful tool that will allow you to transform regular forms into a powerful interactive form.

These are the input fields supported by the form validator. You can extend the BaseInputField class yourself if you would like to make your own, or make use of the CustomInputField.

  • Text Input
  • Integer Input
  • Double Input
  • Selector Input
  • Group
  • Checkbox
  • Custom Input
  • Password Input
  • Match Input

The main parts to the form validator are the input fields and the buttons. Input fields are form inputs that have minimum requirements in order to be valid. For example an Integer input field can require that a non negative integer value be entered. If double is entered, it is considered invalid.

Buttons are controlled by the form validator as well. If not in checkOnlyOnSubmit mode, the buttons controlled by the form will be disabled if 1 input is not valid. These buttons will enable once all inputs are considered valid.

Even if the buttons somehow get enabled, the form validator will prevent the form from being submitted until all inputs are valid.

To show whether fields are valid or invalid, the form validator uses Indicators. There are correct and incorrect indicators. Incorrect indicators show that a field needs to be filled in, an example of this may be a * or a red coloring of the field. Correct indicators show that a field has been correctly filled in. An example of this may be a check mark, or a green coloring of the field.

These indicators, correct and incorrect, are completely customizable and can be anything. They can also be dynamically created by the form validator for you, or can be any existing dom element. In the examples below they will be a red exclamation mark, and a green checkmark.

There are two ways to setup for your validator. You can use json, or it can be done inline in the HTML tags. Here the JSON way will be shown, and then the inline way. With the exception of a couple features, JSON and inline methods offer the same functionality. However best practice is to use JSON for all standard form validation, while inline is best for dynamic forms where the number of inputs is not certain.

Group

JSON

A group is used to group multiple fields together under 1 validation where a specified number of the inputs must be filled in correctly. An example would be a form where the user must fill in their Date of birth, or their Age.

A group will have a validator and indicator (if you want it to) and the group will be valid only when the minimum number of fields have been filled in correctly. However, if one field is filled in with incorrect input, the group will be considered invalid even if there are sufficient number of other input fields filled in already.

Custom Input

JSON

This is the catch all form inputs which require more than the standard set of fields provided by the form validator. For this type of input, you supply your own javascript object which has a isValid function that returns true or false. It can also optionally implement an enable, disable, turnOn, turnOff, clear, and isEmpty functions as well.

Inline

Inline only supports custom input if your validator object can be referred to from an object called validatorGlobal in the global scope. JSON

This input field will enforce a minimum strenght for password entry. The form validator comes with 3 standard levels of password strength, weak, good and strong. In this input, specifiy the minimum level of strength, min being weak, med being good, and max being strong. You can also specify html elements that will appear for each level of strength.
